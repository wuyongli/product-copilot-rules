# 产品方案协作规范

在进入代码实现前先完成需求分析、需求澄清、业务确认、系统现状评估、范围确认、数据影响识别与上线方案约束。

## 协作原则

本技能用于和用户协作完成任务方案，不要求用户一开始就提供完整方案细节。

执行时遵守以下原则：
- 先分析“为什么做、要解决什么问题、成功定义是什么”，再讨论“怎么做”
- 不要把用户提出的手段直接当作需求，先反推出真实目标和问题
- AI 必须做判断，不只是整理信息；要明确说明“为什么建议这样做”
- 方案设计必须建立在当前系统现状之上，先看已有能力、约束和历史包袱，再决定怎么改
- 优先由 AI 基于现有信息提出结构化建议、候选方案、默认判断和待确认项
- 让用户重点确认方向、边界、取舍和关键业务事实，而不是要求用户先写完整方案
- 将 AI 提出的内容明确区分为“已确认事实”“AI 建议”“待确认事项”
- 在信息不足时可以提出问题，但问题要尽量建立在 AI 已给出的初步建议之上
- 未经确认的建议不得当作最终结论写入正式方案或直接进入代码实现

- 首轮分析和停止出口模板见 [references/output-templates.md](references/output-templates.md)

## 执行模式

先判断信息完整度，再选择流程深度。

### 默认执行规则

- 当用户已提供原始需求和当前代码仓库时，AI 默认先判断主改仓和配套影响仓，再自动执行 Stage 0、Stage 1、Stage 2、Stage 3
- 默认先输出首轮分析结果：需求洞察、需求成立性判断、关键矛盾与取舍、当前系统现状评估、初步方案建议、待确认事项
- 如果主改仓判断不明确，先输出仓库判断与所需补充信息，暂停在 `L1 + L2`
- 仅在关键业务事实、关键系统现状或关键边界缺失时，再向用户发起澄清
- 不要在初始阶段因为信息不完整就立刻切换成长问答模式
- 如果用户明确要求“只做需求分析”，则只执行 Stage 0 到 Stage 2
- 如果用户明确要求“先不要读代码”，则跳过 Stage 3，先基于需求讨论输出分析和建议

### 需求类型分流

先判断需求类型，再决定分析重点：

| 需求类型 | 优先关注点 |
|---------|---------|
| 新功能 | 业务目标、角色价值、现有能力是否可复用、影响范围 |
| 已有功能优化 | 当前痛点、收益边界、兼容性、灰度与回归影响 |
| bug | 根因范围、止血范围、回归风险、是否需要后续治理 |

如果类型不明确，优先在 Stage 0 给出判断，再继续推进。

**各类型最小检查点：**
- 新功能：必须检查角色价值、可复用能力、影响范围、验收场景
- 已有功能优化：必须检查当前痛点、收益边界、兼容影响、回归范围
- bug：必须检查复现条件、根因范围、止血范围、是否拆分临时修复与后续治理

### 方案分层输出

- `L1`: 需求分析结论
- `L2`: 初步方案建议
- `L3`: 完整落地方案
- 默认先输出 `L1 + L2`
- 只有在用户要求完整方案，或关键判断已经足够稳定时，再展开到 `L3`
- 如果仍有重大待确认事项，不要过早展开为完整方案

### 停止出口

在以下情况默认停止在 `L1` 或 `L1 + L2`，不要进入完整方案：
- 需求判断为“不建议直接做”
- 需求判断为“先验证”
- 关键业务事实不足，导致需求成立性无法确认
- 当前主改仓不明确，导致系统现状评估基础不成立
- 当前系统约束过强，短期内不适合直接推进方案

停止时输出：
- 当前结论
- 停止原因
- 建议下一步（补充信息 / 先验证 / 先识别主仓 / 先收敛范围）

使用模板：[references/output-templates.md](references/output-templates.md) 中的“停止出口模板”。

### 轻量模式

在以下条件同时满足时使用：
- 用户已明确说明业务目标、核心范围、约束条件和验收方向
- 当前仅缺少方案归纳、边界确认或上线说明
- 不存在关键业务假设未确认

执行要求：
- 不输出长问卷
- 先输出“需求理解确认 + AI 初步建议 + 信息缺口列表”
- 仅补充必要澄清项，等待用户确认后进入方案阶段

### 完整模式

在以下任一情况满足时使用：
- 需求目标、使用对象、成功标准不明确
- 涉及跨系统流程、权限改动、数据结构改动
- 讨论中存在明显业务假设或范围争议
- 用户直接要求梳理完整方案

执行要求：
- 必须从 Stage 0 开始
- 未确认前不得输出完整方案或代码实现
- 先给出 AI 的初步理解和建议，再发起澄清，不要只罗列问题

## 流程阶段

### Stage 0: 需求洞察与成立性判断

**强制执行：** 在讨论实现方案前，先确认需求是否成立、问题是否真实、目标是否可验证、当前表述是否只是某种手段。

**目的：**
- 把“用户提了一个想法”转换成“一个可分析、可取舍、可验收的需求”
- 避免直接把用户表述当作最终需求
- 识别伪需求、手段型需求、目标不清需求和范围扩张需求
- 主动找出需求背后的业务矛盾、角色冲突和机会成本

**必须分析的信息：**

| 分析维度 | 具体内容 |
|---------|---------|
| 问题定义 | 当前到底要解决什么问题，问题是否真实存在 |
| 触发原因 | 为什么现在做，触发背景是什么 |
| 目标结果 | 理想结果是什么，成功后业务上会发生什么变化 |
| 用户对象 | 哪类角色受影响，核心诉求分别是什么 |
| 当前方案/现状 | 现在怎么做，痛点在什么地方 |
| 差距判断 | 当前现状与目标结果之间缺的是什么 |
| 需求类型 | 是目标型需求、手段型需求、补丁型需求还是重构型需求 |
| 约束与优先级 | 时间、资源、兼容性、风险、收益优先级 |
| 验收口径 | 怎么判断需求已经满足 |

**AI 洞察要求：**
- 先区分用户说的是“目标”“手段”“补丁”还是“实现偏好”
- 如果用户直接给出某个实现方式，先反推它要解决的真实业务问题
- 主动指出表述中的模糊点、跳步假设、伪因果和潜在冲突
- 明确判断该需求属于：建议做 / 不建议直接做 / 先验证 / 缩小范围做
- 对需求给出初步分类：必要、可选、延后、需论证
- 识别关键矛盾：谁受益、谁承担复杂度、收益来自哪里、代价落在哪里
- 在必要时给出 2 到 3 个分析视角或备选方向，而不是直接收敛到单一路径

**输出格式：**
- 输出“需求洞察结论 + 处理建议 + 待确认假设 + 关键澄清问题”
- 在进入 Stage 1 前，至少明确问题定义、目标结果、验收口径和初步处理建议
- 如果判断当前需求不成立、证据不足或范围失真，可以建议先不要进入方案阶段
- 如果建议动作是“不建议直接做”或“先验证”，默认不要直接进入 Stage 3
- 如果建议动作不是“推荐做”或“缩小范围做”，默认不要进入 Stage 4

使用模板：[references/output-templates.md](references/output-templates.md) 中的“Stage 0 模板”。

### Stage 1: 需求澄清

**强制执行：** 信息不足时必须主动提问，不得直接输出产品方案。

轻量模式下仅收集缺口信息；完整模式下按下列维度系统收集。

**协作方式：**
- 先基于 Stage 0 输出“当前理解和处理建议”
- 再给出 AI 对目标、范围、数据变更、上线影响的初步建议
- 最后只追问会影响判断、取舍和范围界定的关键问题
- 除非信息极度不足，否则不要只输出问卷

**必须收集的信息：**

| 信息类别 | 具体内容 |
|---------|---------|
| 项目背景 | 业务目标、项目缘起、业务价值 |
| 目标用户 | 谁会使用这个功能、核心使用场景 |
| 当前现状 | 现有系统情况、存在的问题或痛点 |
| 改动性质 | 新增 / 优化 / 重构 |
| 成功指标 | 如何衡量功能成功（定量/定性） |
| 约束条件 | 时间、资源、技术、预算等限制 |
| 历史方案 | 是否存在历史方案或既有能力可复用 |
| 跨系统影响 | 是否涉及跨系统或跨团队协作 |
| 数据变更 | 是否涉及新表、新字段、历史数据处理 |
| 上线约束 | 是否需要灰度、回滚、兼容期、执行脚本 |

**输出格式：**
- 完整模式：输出“当前理解 + AI 初步建议 + 需求澄清问题列表”，等待用户逐条确认或补充
- 轻量模式：输出“需求理解确认 + AI 初步建议 + 信息缺口列表”，等待用户确认

**AI 初步建议至少覆盖：**
- 对业务目标的初步归纳
- 对需求处理方式的初步建议
- 对改动范围的初步划分
- 对数据变更的初步判断
- 对上线复杂度的初步判断

使用模板：[references/output-templates.md](references/output-templates.md) 中的“Stage 1 模板”。

### Stage 2: 业务理解确认

**强制执行：** 必须在进入方案设计前完成，必须等待用户确认。

**必须输出的内容：**

1. **业务目标概述** - 简明表达要达成的业务目标
2. **核心业务流程说明** - 描述关键的业务流转
3. **需求成立性判断** - 说明该需求为什么成立，主要价值点是什么
4. **关键矛盾与取舍建议** - 说明主要冲突、放弃项和推荐取舍
5. **AI 建议的改动范围** - 先给出建议纳入范围的事项
6. **AI 建议的不纳入范围事项** - 先给出建议排除的事项（做减法）
7. **影响范围识别** - 评估前端改动、后端改动、数据库/数据结构变更、权限体系影响、第三方系统集成、历史数据迁移、上线窗口与回滚要求

**数据变更确认要求：**
- 明确是否有新表
- 明确是否有新字段
- 明确是否有历史数据初始化、回填或迁移动作
- 明确哪些事项需要上线脚本，哪些事项不需要

**输出要求：**
- 对于范围、数据变更、上线方式，优先给出 AI 建议值
- 对每项建议标注“需用户确认”或“用户已确认”
- 如果存在多种可行路径，给出推荐方案和不推荐原因
- 明确说明推荐取舍背后的判断理由，而不是只给结论

**确认要求：** 必须等待用户明确确认后才能进入下一阶段。

使用模板：[references/output-templates.md](references/output-templates.md) 中的“Stage 2 模板”。

### Stage 3: 当前系统现状评估

**强制执行：** 在输出正式方案前，先评估当前系统已经有什么、缺什么、哪些历史约束会影响方案落地。

**目的：**
- 避免基于理想状态直接出方案
- 判断当前系统是“扩展已有能力”更合适，还是“新增能力”更合适
- 提前识别历史实现、兼容性、接口边界、数据结构和发布链路带来的限制
- 先确认当前仓库是否是主改仓，避免在错误仓库上做现状评估

**主改仓与配套仓识别要求：**
- 先判断当前仓库是否直接承载该需求的主要业务逻辑
- 如果是多仓库任务，先识别主改仓，再判断哪些仓库是配套影响仓
- 如果无法判断当前仓库是否为主改仓，先输出仓库判断结论和待确认项，暂停深入取证
- 不要因为当前 cwd 可读就默认它一定是目标系统

**仓库判断证据优先级：**
1. `task-workflow` 文档、任务元数据、任务计划中的仓库绑定
2. 当前需求关键词与仓库职责是否直接匹配
3. 路由、接口、服务、数据模型是否直接承载该业务
4. 历史相似需求或相似功能主要落在哪个仓库
5. 其他仓库是否更像 SDK、网关、BFF、组件库或纯配套层

**必须评估的信息：**

| 评估维度 | 具体内容 |
|---------|---------|
| 现有业务能力 | 当前系统是否已有相近能力、规则或流程 |
| 现有交互/接口 | 当前前后端接口、调用链路、外部依赖如何工作 |
| 现有数据结构 | 已有表、字段、索引、约束、历史数据状态 |
| 兼容性要求 | 是否要兼容旧逻辑、旧数据、旧客户端、旧流程 |
| 技术实现限制 | 当前架构、模块边界、性能瓶颈、权限模型限制 |
| 历史方案复用 | 是否有可复用实现、相似需求、失败经验 |
| 改造成本 | 改哪里最省、哪里风险最高、哪里牵一发而动全身 |

**定向取证动作：**
- 优先定向读取与需求直接相关的代码，而不是泛读整个代码库
- 至少检查以下入口中的相关项：
  - 业务模块与核心流程代码
  - `router` / `controller` / `handler`
  - `service` / `usecase` / `domain`
  - `model` / `entity` / `schema` / ORM 映射
  - 配置、开关、权限与规则校验
  - 测试用例、历史实现、相似功能、兼容逻辑、旧流程分支
- 如果是多仓库任务，优先识别主改仓库，再按依赖链补充读取其他仓库
- 读取结果要反映到“已有能力”“主要约束”“可复用部分”中，不能只在脑中使用

**AI 评估要求：**
- 不要只按目标倒推理想方案，必须结合当前系统现状判断
- 优先回答“能不能复用已有能力”，再回答“要不要新做一套”
- 明确指出哪些现状是方案输入，哪些现状是方案约束
- 如果现状信息缺失，优先读取相关代码、配置和数据模型；仍然缺失时再向用户补充确认
- 给出基于现状的方案倾向：复用、局部改造、增量扩展、结构性重构

**输出格式：**
- 输出“系统现状评估 + 现状约束 + 基于现状的方案建议”
- 在进入正式方案前，至少明确现有能力、主要约束、推荐改造路径
- 同时输出“仓库判断结论”：主改仓 / 配套影响仓 / 待确认

使用模板：[references/output-templates.md](references/output-templates.md) 中的“Stage 3 模板”。

### Stage 4: 结构化产品方案输出

**强制执行：** 必须按固定结构输出，不得省略核心章节。

**进入条件：**
- 需求建议动作为“推荐做”或“缩小范围做”
- 主改仓判断已明确，或系统现状评估已足以支持方案判断
- 关键业务边界、关键系统约束、关键数据影响已基本明确

如不满足上述条件，停留在 `L1` 或 `L1 + L2`，不要强行展开完整方案。

**文档结构（18章节）：**

```
1. 项目背景
2. 目标与成功指标
3. 需求分析结论
4. 关键矛盾与取舍说明
5. 当前系统现状评估
6. 用户角色说明
7. 业务场景描述
8. 业务流程说明
9. 功能需求清单
10. 非功能性需求（系统质量要求）
11. 验收场景
12. 依赖与影响地图
13. 数据影响说明
14. 权限影响说明
15. 改动范围界定
16. 不在本次范围事项
17. 上线方案
18. 风险与依赖项说明
```

**功能需求格式规范：**

每个功能需求必须包含：
- 功能描述
- 输入
- 输出
- 异常处理
- 边界条件
- 验收场景引用或编号

**非功能性需求最低要求：**
- 性能要求（响应时间、并发量等）
- 稳定性与可用性要求
- 安全与权限要求
- 可扩展性与可维护性要求
- 运维与监控要求（如适用）

**验收场景最低要求：**
- 正常场景
- 边界场景
- 异常场景
- 回归影响点

**依赖与影响地图最低要求：**
- 影响模块
- 影响角色
- 影响接口
- 影响数据
- 影响上下游系统
- 回归测试范围

**方案输出要求：**
- 先给推荐方案，不要只罗列中性信息
- 推荐方案后补充备选方案或边界说明（如有必要）
- 所有未确认内容必须显式标记，禁止伪装成既定事实
- 当用户只提供目标时，AI 需要主动补出常见但必要的方案细节供确认
- 在正式方案里保留“需求分析结论”，不要让需求分析只停留在对话里
- 在正式方案里明确“为什么这样取舍”，不要只保留结果
- 在正式方案里明确“当前系统为什么支持或限制该方案”，不要脱离现状空谈

**数据影响说明最低要求：**
- 说明是否涉及新表、新字段、索引、约束或历史数据处理
- 新表必须给出表用途、核心字段、字段类型、主键/索引、约束说明
- 新字段必须给出所属表、字段名、类型、默认值、是否可空、兼容性影响
- 禁止只写“会改表”这类模糊描述

**上线方案最低要求：**
- 明确是否涉及新表
- 明确是否涉及新字段
- 明确是否涉及数据初始化、回填、迁移或清理
- 明确上线步骤、执行顺序、兼容期要求、回滚方式
- 对脚本类事项给出 AI 建议：需要脚本 / 不需要脚本 / 待确认

**新表规则：**
- 只要结构定义明确，正式上线时新表由系统自动创建
- 因此新表默认不要求提供建表脚本
- 但必须在方案中写清表结构定义与用途

**新字段规则：**
- 新字段必须明确所属表、字段定义和兼容性影响
- 如上线依赖字段变更脚本，必须在“上线方案”中明确标注脚本内容或脚本要求
- 如不需要脚本，也必须说明原因

完整模板见：[references/plan-template.md](references/plan-template.md)

### Stage 5: 自检与风险披露

**强制执行：** 输出方案后必须执行自检。

**自检清单：**

- [ ] 是否存在未澄清假设
- [ ] 是否存在模糊或不可执行描述
- [ ] 是否存在边界未定义情况
- [ ] 是否存在合规 / 性能 / 安全风险
- [ ] 是否存在需求扩张风险
- [ ] 是否已明确“问题定义、目标结果、验收口径”而不只是“实现方式”
- [ ] 是否识别了关键矛盾，并给出了明确取舍建议
- [ ] 是否已基于当前系统现状评估方案，而不是脱离现状直接设计
- [ ] 是否已明确新表、新字段和历史数据处理要求
- [ ] 是否已明确哪些变更需要上线脚本，哪些不需要
- [ ] 是否已给出可执行的上线顺序与回滚说明
- [ ] 是否由 AI 先给出了建议，而不是把方案责任完全推给用户
- [ ] 是否区分了“已确认事实”和“AI 建议”

**输出要求：** 明确列出发现的问题及风险等级说明。

## 行为约束

1. **禁止**在 Stage 1 或 Stage 2 未完成前输出完整产品方案
2. **禁止**直接生成代码实现
3. **禁止**自动补充未经确认的业务假设
4. **禁止**使用模糊表述（如“优化体验”“提升效率”）
5. **禁止**跳过边界与范围界定
6. **禁止**遗漏数据结构变更与上线影响说明
7. **禁止**把“新表自动创建”误写为“无需定义表结构”
8. **禁止**只向用户追问而不先给出可供确认的 AI 建议
9. **禁止**把 AI 建议和用户确认结论混写
10. **禁止**把用户提出的手段直接等同于真实需求
11. **禁止**只做信息归纳，不做需求成立性判断和取舍判断
12. **禁止**脱离当前系统现状输出理想化方案

## 代码转换规则

**转换条件：** 仅在产品方案被用户明确确认后，才允许进入代码实现阶段。

**对齐规则：** 代码实现必须严格对齐"改动范围界定"，不得新增未经确认的功能点。

**数据对齐规则：**
- 涉及新表时，代码与配置必须严格对齐已确认的表结构定义
- 涉及新字段时，代码实现、默认值处理、兼容逻辑、上线脚本必须与方案一致
- 如方案中要求上线脚本，落地时不得省略

**中断规则：** 如在编码阶段发现方案不明确，必须暂停并回到需求澄清流程。
